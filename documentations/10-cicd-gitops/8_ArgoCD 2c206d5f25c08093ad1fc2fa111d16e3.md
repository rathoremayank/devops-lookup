## 2. Introduction

### Overview of Argo CD

Argo CD is a **declarative, GitOps-continuous deployment tool** for Kubernetes. It automates the deployment of applications by continuously synchronizing the desired state defined in Git repositories with the actual state running on Kubernetes clusters.

### Why It Matters in Modern DevOps Platforms

Argo CD addresses critical challenges in cloud-native deployments:

1. **Single Source of Truth**: All configurations, manifests, and deployments are version-controlled in Git
2. **Audit & Traceability**: Every change has a Git commit history with author, timestamp, and rationale
3. **Automated Reconciliation**: Continuous drift detection and automatic correction ensure consistency
4. **Multi-Cluster Management**: Centralized control over multiple Kubernetes clusters from a single Argo CD instance
5. **GitOps at Scale**: Enables platforms to manage hundreds of applications across teams without manual kubectl commands
6. **Disaster Recovery**: Git repository acts as a complete backup; clusters can be rebuilt from Git state

### Real-World Production Use Cases

1. **Microservices Platform Bootstrapping**
   - Deploy entire cluster infrastructure (CRDs, operators, observability stack) with a single Argo CD Application
   - App of Apps pattern provisions 50+ platform services automatically

2. **Multi-Tenant SaaS Deployments**
   - Isolate teams and environments using AppProjects with namespace/cluster boundaries
   - Different teams manage their own applications while platform team controls access policies

3. **Production-Grade Progressive Delivery**
   - Integration with Argo Rollouts for canary deployments with automated metric analysis
   - Zero-downtime blue/green deployments with instant rollback capability

4. **Disaster Recovery & Cluster Failover**
   - Git as the single source of truth enables rapid cluster recreation
   - Geo-distributed deployments across multiple cloud providers (AWS, Azure, GCP)

5. **Compliance & Regulated Environments**
   - Immutable audit trail of all deployments
   - RBAC with SSO integration for access control
   - Support for sealed secrets and vault integration for secret management

### Where It Typically Appears in Cloud Architecture

- **Platform Layer**: Core infrastructure orchestration (runs in production Kubernetes cluster)
- **Multi-Cluster Management**: Central control plane managing edge, regional, and cloud clusters
- **CI/CD Pipeline Integration**: Webhook triggering from Git repositories or custom CI systems
- **GitOps Hub**: Central deployment engine for all environments (dev, staging, prod, DR)
- **Service Mesh Integration**: Works alongside Istio/Linkerd for advanced traffic management
- **Observability Stack**: Integrates with Prometheus, Grafana, and logging systems for health monitoring

## 3. Foundational Concepts

### Key Terminology

**GitOps**
- Operational pattern where Git repositories serve as the single source of truth for system state
- All changes flow through Git (commits, PRs, merges), not direct infrastructure modifications
- Continuous reconciliation ensures live state matches declared state

**Application (Argo CD)**
- A Kubernetes CRD defining *what to deploy*, *from where*, and *to which cluster*
- Not the workload itself, but an *intent* for deploying a workload
- Includes source (Git repo), destination (cluster/namespace), sync policy, and project

**Synchronization**
- Process of applying Git-declared manifests to a Kubernetes cluster
- Can be manual (requires human approval) or automated (continuous)
- Includes diffing (comparison), filtering (AppProject restrictions), and applying (kubectl apply behavior)

**Drift Detection**
- Continuous comparison of Git desired state vs. live cluster state
- Identifies when cluster has been modified outside of GitOps (manual kubectl, API calls, webhooks)
- Triggers alerts, syncs, or automated corrections based on policy

**Reconciliation**
- The loop that ensures cluster state matches Git state
- Default 3-minute interval, but can be triggered by webhooks or manual commands
- Core mechanism that makes GitOps continuous and not just one-time deployments

### Architecture Fundamentals

**Three-Tier Architecture**

1. **API Server (Frontend)**
   - Exposes gRPC and REST APIs
   - Serves web UI
   - Handles authentication and RBAC enforcement
   - Acts as proxy between users and core components

2. **Repository Server (Manifest Rendering)**
   - Fetches and caches Git repositories
   - Renders manifests (Helm, Kustomize, Jsonnet, plain YAML)
   - Offloads CPU-heavy templating work from controller
   - Horizontally scalable for performance

3. **Application Controller (Core Logic)**
   - Watches Application CRDs and continuously reconciles state
   - Performs diffing (Git vs. cluster)
   - Applies manifests and manages resource lifecycle
   - Tracks resource health and sync status
   - Implements sync waves, hooks, and pruning logic

**Git as the Single Source of Truth**
- All infrastructure and application configurations reside in Git
- Cluster state is derived from Git, not the inverse
- Changes to live cluster are considered "drift" and should be corrected

**Declarative vs. Imperative**
- Argo CD is **declarative**: you declare desired state in Git
- Argo CD handles reconciliation automatically
- Never use `kubectl apply` in production; changes flow through Git only

### Important DevOps Principles

**Immutability**
- Git commits are immutable and auditable
- Every deployment has a traceable commit hash
- Enables reproducibility and blame/accountability

**Least Privilege Access**
- Users should only have permissions for their scope (namespace, cluster, project)
- Separate roles for different operational tasks (sync, create, read)
- RBAC and AppProjects enforce boundaries

**Consistency Across Environments**
- Same manifests (with overlays) for dev/stage/prod reduces "it works on my machine" problems
- Promotes monorepos or structured polyrepos with unified standards
- Drift detection ensures consistency is maintained

**Automation & Human Control Balance**
- Auto-sync enables velocity but requires strong governance
- AppProjects + RBAC prevent rogue deployments
- Manual sync for critical workloads with strong validation gates

**Observable & Auditable**
- Web UI and CLI provide visibility into all deployments and changes
- Webhook integrations for monitoring and alerting
- Complete audit trail: who, what, when, why (via Git commits)

### Best Practices

1. **Repository Organization**
   - Use clear directory structure: `apps/`, `infrastructure/`, `overlays/`
   - Separate concerns: infrastructure CRDs, platform components, application workloads
   - Use sync waves to order deployments (CRDs → operators → workloads)

2. **Sync Policy Strategy**
   - Manual sync for production, stateful systems, and major infrastructure changes
   - Automated sync for dev/staging and stateless application microservices
   - Always enable `prune` for true GitOps declarativeness
   - Use `selfHeal` carefully; it overwrites manual cluster changes

3. **Secret Management**
   - Never commit plaintext secrets to Git
   - Use encryption (SOPS, Sealed Secrets) or external vaults (Vault, ESO)
   - Cluster credentials stored as Kubernetes Secrets, not in Git

4. **Multi-Tenancy & Isolation**
   - Use AppProjects to enforce namespace and cluster boundaries
   - Separate Git repos per team or shared monorepo with strict RBAC
   - Leverage per-environment overlays to avoid duplication

5. **Monitoring & Alerting**
   - Track sync success/failure rates and duration metrics
   - Alert on applications stuck in OutOfSync status
   - Monitor repo-server latency and manifest size
   - Health checks for dependent services

### Common Misunderstandings

**Myth: "Argo CD is a CI tool"**
- Reality: Argo CD is a **deployment** tool, not a build/test tool
- CI tools (Jenkins, GitHub Actions) build and test; they push artifacts and trigger Argo CD
- Argo CD owns the deployment and reconciliation, not the build pipeline

**Myth: "GitOps means no manual kubectl"**
- Reality: Manual kubectl is allowed but creates drift
- If you use manual kubectl, you must sync to restore Git state
- The principle is: Git should be your first place to make changes

**Myth: "Argo CD auto-sync means hands-free deployment"**
- Reality: Auto-sync requires:
  - Strong CI/CD gates (tests, security scans)
  - Tight RBAC and AppProject controls  
  - Monitoring and alerting for failures
  - Strategy for rollbacks

**Myth: "One Argo CD instance can't scale"**
- Reality: Horizontal scaling of repo-server and controller handles 100s–1000s of applications
- Multi-cluster architectures scale to thousands of apps across scores of clusters

**Myth: "Argo CD replaces infrastructure provisioning tools"**
- Reality: Argo CD deploys *to* infrastructure
- Terraform/CloudFormation provision cloud resources (VPCs, clusters, storage)
- Argo CD then manages applications *inside* those provisioned clusters

# **Argo CD Architecture**

Argo CD follows a **controller-based GitOps architecture**, built around Kubernetes CRDs and control loops. At a high level:

- **Git is the source of truth**
- Argo CD continuously **compares Git → Live cluster**
- A set of components implement API access, Git access, reconciliation, SSO/RBAC, and manifest caching

Below is the breakdown of each architectural component **exactly the way interviewers expect**.

---

## **1. API Server**

### **What it is**

`argocd-server` is the **front-end** of Argo CD. Everything (UI, CLI, API, SSO) goes through this service.

### **Key Responsibilities**

- Exposes **gRPC & REST APIs**
- Serves the **Web UI**
- Handles **authentication** (Dex/OIDC)
- Enforces **RBAC** (who can see/sync what)
- Proxies operations to:
    - Application Controller
    - Repo Server

### **Critical Functions**

- **Cluster access**: provides credentials for different target clusters
- **Token management**: JWTs, session tokens, CLI tokens
- **App CRUD**: creating, updating, deleting Applications (CRDs)
- **Access control**: AppProject-based and RBAC enforcement

### **Why it exists**

You DO NOT want direct access to repo-server or controller. API server isolates concerns and applies security.

---

## **2. Repo Server**

### **What it is**

The repo-server is responsible for:

- Fetching Git repos
- Rendering manifests (Helm, Kustomize, Jsonnet, KSOPS)
- Caching rendered output for speed

### **Key Responsibilities**

- Git clone, Git fetch (read-only)
- Decrypt (optional: SOPS with KSOPS plugin)
- Render manifests using:
    - **Kustomize**
    - **Helm**
    - **Jsonnet**
    - **Plain YAML**
- Provide **manifests to the Application Controller**
- Maintain **in-memory cache** to avoid repeated rendering

### **Why it's separate**

Rendering manifests can be expensive. Repo-server offloads CPU-heavy work from the controller. This separation allows horizontal scaling.

### **Scaling Tip**

If you have massive Helm charts or heavy Kustomize overlays:

→ **Scale repo-server** (horizontal)

→ Enable **manifest caching** (default)

---

## **3. Application Controller**

### **What it is**

The brain of Argo CD. A Kubernetes controller that continuously reconciles **desired (Git) → live (cluster)** state.

### **Key Responsibilities**

- Compare Git vs cluster
- Detect drift
- Make the app **Healthy/OutOfSync/Synced**
- Apply manifests (kubectl apply behavior using Kubernetes API)
- Prune resources
- Run sync hooks
- Watch resources for health signals
- Self-heal live drift (if enabled)

### **How it works internally**

- Watches `Application` CRD objects
- For each app:
    1. Ask repo-server for manifests
    2. Compare with live cluster state
    3. Generate diffs
    4. Apply or prune resources based on sync policy

### **Important**

Application Controller NEVER talks to Git directly. It always goes through repo-server.

---

## **4. Dex / SSO Integration**

### **Why Dex?**

Dex is an OIDC identity provider used by Argo CD to integrate external identity providers.

### **Supported Auth Methods**

- GitHub
- GitLab
- Google
- Microsoft AD
- Okta
- LDAP (via connectors)
- SAML-based IdPs (via connectors)

### **Flow**

1. User clicks “Login with SSO”
2. Dex redirects to Identity Provider
3. User authenticates
4. Dex returns ID token to API server
5. Argo CD applies **RBAC based on groups/claims**

### **Key Interview Points**

- Dex is **optional**.
    
    Argo CD supports **direct OIDC** without Dex in newer versions.
    
- Dex provides a unified interface when integrating multiple IdPs.

---

## **5. Redis (for caching manifests & state)**

### **Why Redis exists in Argo CD**

Argo CD uses Redis to speed up operations via:

- Caching **manifest generation results**
- Tracking **App state**
- Caching **live cluster resources**
- Storing **diff results**

### **Where it helps**

- Large orgs with:
    - heavy Helm charts
    - many Applications (500+)
    - multi-cluster setups

### **Behaviour**

- Redis is optional but heavily recommended for performance.
- Without Redis:
    - repo-server re-renders everything all the time
    - more CPU
    - slower sync cycles

---

## **6. How Diffing Works**

This is a crucial interview topic.

### **Argo CD diff = Desired (Git manifest rendered) vs Live (cluster)**

### Steps:

1. Repo-server renders desired manifests
2. Application Controller fetches live resources from the cluster
3. Diff engine compares:
    - Spec differences
    - Missing resources
    - Extra resources (to prune)
    - Metadata drift
    - Immutable field changes

### **Advanced Behaviors**

- Diff ignores:
    - Some autogenerated fields (managedFields, timestamps)
    - Some metadata based on ignore rules
- Supports:
    - **Custom diffs using Lua**
    - **IgnoreDifferences** spec to ignore specific fields

### **Common Interview Question**

**“How does Argo CD diff differ from kubectl diff?”**

Answer:

- Argo CD considers cluster state + Git + AppProject restrictions.
- It also understands Helm/Kustomize structures.
- Supports pruning logic.

---

## **7. Reconciliation Loop (default: 3 minutes)**

### **What it is**

A periodic loop inside Application Controller that:

- Checks each Application every **3 minutes** (default)
- Detects drift by comparing Git vs live
- Emits updated sync/health status

### **Triggers for reconciliation**

- Time-based loop (3m)
- Webhooks from Git repos (optional)
- Manual `argocd app sync`
- CLI/UI triggered refresh
- Changes in:
    - Application CRD spec
    - AppProject spec
    - Target cluster resources

### **Self-Heal**

If **automated sync + selfHeal=true**, reconciliation will auto-apply fixes when:

- Someone manually modifies cluster objects
- Drift is detected

### How to reduce reconciliation load

- Increase refresh timeout for huge clusters
- Or use webhooks to do event-driven reconciliation

# **2. Core Concepts – Detailed Study Material**

---

## **1. Applications (The Core Argo CD CRD)**

An **Application** is the fundamental Argo CD object.

It defines **what to deploy**, **from where**, **to which cluster**, and **with which tool (Helm/Kustomize/etc.)**.

### **Key Fields**

- **source:**
    - repoURL
    - path
    - revision (branch/tag/commit)
    - plugin/helm/kustomize/jsonnet options
- **destination:**
    - namespace
    - server (cluster API endpoint or Kubernetes context)
- **syncPolicy:**
    - automated/manual
    - prune
    - selfHeal
    - replace
- **project:**
    
    Links to an AppProject for RBAC, cluster whitelisting, namespace restrictions.
    

### **How Argo CD treats an Application**

- Renders manifests (via repo-server)
- Compares desired vs live cluster
- Syncs resources
- Tracks health of resources
- Tracks drift

### Key Interview Note:

**An Application is NOT your workload. It is a controller-managed *intent* for your workload.**

---

## **2. App of Apps Pattern**

This is a common Argo CD design pattern used for **bootstrapping entire environments**.

### **How it works**

- You create a **parent Application**.
- That parent points to a Git directory containing MANY Application YAMLs.
- Argo CD creates child Applications automatically.

### **Use Cases**

- Cluster bootstrapping (install Argo CD itself, ingress, monitoring, CI runners, etc.)
- Multi-team setups
- Environment templates (dev/stage/prod)
- GitOps for multi-cluster environments

### **Benefits**

- Declarative environment composition
- Scalable: 1 parent → 50+ child apps
- Easy cluster onboarding: just apply the root Application

### **Interview-Ready Explanation**

**“App of Apps allows you to version-control not only workloads but the GitOps architecture itself. Argo CD becomes fully self-managing.”**

---

## **3. Projects (AppProject)**

AppProject is the **security/organization boundary** in Argo CD.

### **Key Features of AppProject**

- Restrict **which repos** an application can use
- Restrict **which clusters** it can deploy to
- Restrict **which namespaces** it can touch
- Define **RBAC** for who can sync/override resources
- Define **sync windows** (allowed/disallowed sync times)
- Define **namespace resource limits**
- Define **cluster-scoped resource whitelist/blacklist**

### **Why it matters**

AppProject prevents:

- rogue teams deploying cluster-scoped objects
- workloads leaking into other namespaces
- unauthorized Git repos being synced into production

### **Interview Point**

**AppProject = a policy boundary defining “what an application is allowed to do”.**

---

## **4. Sync Statuses**

These indicate **desired vs live cluster state** from GitOps perspective.

### **Synced**

- Live cluster matches desired Git state.
- No drift.
- No pending operations.

### **OutOfSync**

Argo CD detected changes in:

- Git repo
- cluster resources
- Helm values
- Kustomize overlays
- missing/extra Kubernetes objects

Common causes:

- manual kubectl apply
- Git changed but sync not triggered yet
- differences in immutable fields

### Interview Tip:

Expect to be asked:

**“What causes OutOfSync even when nothing changed?”**

Answer:

- Metadata drift (annotations/labels)
- Fields ignored by diff settings
- CRDs introducing new default fields
- Missing ownerReferences

---

## **5. Health Statuses**

Argo CD evaluates resource health using internal logic + custom health checks.

### **Healthy**

All resources meet their ready conditions:

- Deployments: desired = ready
- StatefulSets: replicas ready
- Services: no checks (always Healthy)
- Ingress: rules admitted

### **Degraded**

Resources failing or stuck:

- Pods crashlooping
- Unready deployments
- Failed jobs
- Image pull errors

### **Missing**

A resource is expected (from Git) but missing in the cluster.

### **Progressing**

Deployments rolling out, Jobs running, StatefulSets updating — basically, “in rollout”.

### **Custom Health Checks (Lua)**

You can define custom health logic for CRDs like:

- Argo Rollouts
- External Secrets
- Kafka operators
- Databases

---

## **6. Sync Waves & Hooks**

A powerful ordering mechanism used during **sync operations**.

---

## **Sync Waves**

Argo CD annotates resources with:

```
argocd.argoproj.io/sync-wave: "5"

```

Lower wave → applied first.

Default wave = 0.

### Example Use Case

- CRDs → wave -1
- Custom operators → wave 0
- Workloads → wave 1
- Ingress → wave 2

---

## **Hooks**

Specially tagged resources that run at different stages:

### **PreSync**

Runs before applying any manifests

Use case: migration jobs, validation scripts

### **Sync**

Executed during the sync

Use case: Helm pre-upgrade behavior but within Argo CD

### **PostSync**

Runs after main manifests

Use case: smoke tests, notification jobs

### **SyncFail**

Runs only when sync fails

Use case: cleanup, rollback triggers

Hooks use annotations like:

```
argocd.argoproj.io/hook: PreSync

```

---

## **7. Resource Tracking**

Argo CD needs to know **which Kubernetes objects belong to which Application**.

Tracking uses:

1. **OwnerReferences** (preferred)
2. **Annotations** (`argocd.argoproj.io/instance`)
3. **TrackingMethod** setting in Application

### Why this matters

If resources are not tracked:

- Argo CD won’t prune them
- Argo CD won't detect drift
- Resources may not be associated with the application

Tracking can break when:

- Helm charts overwrite metadata
- CRDs drop labels/annotations
- Cluster mutating webhooks mutate metadata

---

## **8. Automated vs Manual Sync**

### **Manual Sync**

Default.

- Argo CD detects OutOfSync
- But waits for human approval
- Used in controlled environments (production)

### **Automated Sync**

Argo CD applies changes automatically.

```
syncPolicy:
  automated: {}

```

Advanced flags:

- prune: true
- selfHeal: true

### **Use Case**

- Dev environments
- Non-critical workloads
- High-speed CI/CD pipelines

### Interview Tip

If you enable automated sync **without** selfHeal, Argo CD:

- Applies Git changes
- BUT does NOT correct manual cluster drift

---

## **9. Sync Options (Prune, SelfHeal, Replace)**

---

## **Prune**

Deletes resources in the cluster that are no longer in Git.

```
syncPolicy:
  automated:
    prune: true

```

Dangerous but essential for GitOps purity.

---

## **SelfHeal**

Continuously reverts manual cluster changes.

```
syncPolicy:
  automated:
    selfHeal: true

```

If someone edits a Deployment manually:

→ Argo CD overwrites it back to Git version.

---

## **Replace**

Use server-side "replace" instead of "apply".

```
syncOptions:
  - Replace=true

```

Why?

- Needed when immutable fields change
- Useful for CRDs and StatefulSets updates

Downside:

- Replace is more disruptive (delete + create)

# **3. Deployment Strategies – Detailed Study Material**

Argo CD itself **does NOT perform Blue/Green or Canary**.

These strategies are implemented via **Argo Rollouts**, which Argo CD manages as a workload.

Your GitOps pipeline → Argo CD → applies Rollout CRDs → Rollout controller performs progressive delivery.

---

## **1. Blue/Green Deployment (Argo Rollouts)**

## **What Blue/Green Means**

Two full versions of the application exist simultaneously:

- **Green** → New version
- **Blue** → Stable/active version serving traffic

Only one receives production traffic at a time.

## **How Argo Rollouts implements Blue/Green**

Argo Rollouts introduces:

- **Rollout CRD**
- **Preview Service**
- **Active Service**

### Example Flow

1. Argo CD applies a new Rollout manifest from Git.
2. Rollouts controller creates a **preview ReplicaSet** (Green).
3. Preview service routes traffic ONLY to the Green version.
4. Operators/testers validate.
5. When approved:
    - Rollouts controller shifts **active service** from Blue → Green.
6. Old Blue version stays for optional quick rollback.

### **Benefits**

- Zero-downtime
- Instant rollback
- Our traffic routing is clean and predictable

---

## **2. Canary Deployment (Argo Rollouts)**

## **What Canary Means**

Only a percentage of traffic is shifted to the new version first.

### Common Steps

- 5% → 20% → 50% → 100%
- Automated or manual promotion

## **How Argo Rollouts implements Canary**

Rollouts integrates with:

- **Ingress Controllers** (NGINX, ALB, GCE)
- **Service Mesh** (Istio)
- **Gateway API**
- **Service-based canary (SVC1/SVC2)**

### Features

- Weighted traffic shifting
- Pause steps
- Analysis templates (automated analysis)
- Abort conditions

### Example Canary Strategy

```
strategy:
  canary:
    steps:
    - setWeight: 10
    - pause: {duration: 1m}
    - setWeight: 50
    - pause: {duration: 5m}

```

Argo Rollouts controller orchestrates traffic shifting.

Argo CD simply applies this YAML.

### **Benefits**

- Gradual exposure
- Real-time rollback on failures
- A/B testing ability
- SLO-driven promotion

---

## **3. Progressive Delivery Basics**

Progressive delivery = Continuous delivery **with safety gates**.

It is delivery with **gradual rollout + automated verification + rollback**.

### Core Concepts

- **Incremental rollout**: Step-by-step exposure
- **Metrics-driven analysis**:
    - Prometheus
    - Datadog
    - Kayenta
- **Automated rollback** on failure
- **Feature gating** (canary release independent of code deploy)
- **Human approval gates** optional

### Key Terms in Rollouts

- **Analysis Template**: Defines metrics checks
- **AnalysisRun**: Execution of analysis
- **Experiments**: Multiple versions tested simultaneously
- **Pause Steps**: Manual or timed holds

### Example Analysis Template

Checks error rate, p95 latency, or custom metrics before progressing.

---

## **4. How Argo CD Ties Into Argo Rollouts**

Argo CD and Argo Rollouts work side-by-side:

### **Argo CD Responsibilities**

- Fetch manifests from Git
- Render Helm/Kustomize templates
- Apply Rollout CRDs into Kubernetes
- Track the health of Rollout objects using **custom health checks**
- Detect drift and sync changes
- Provide UI visualization for Rollouts

Argo CD does NOT:

- Shift traffic
- Perform progressive steps
- Run analysis or pause logic
- Roll back traffic routing

### **Argo Rollouts Responsibilities**

- Manage ReplicaSets for canaries/blue-green
- Integrate with ingress/service mesh for traffic shifting
- Handle analysis and automated promotion
- Roll back on failure
- Emit detailed rollout events/statuses

This separation is intentional.

---

## **5. What Argo CD Shows in the UI for Rollouts**

Argo CD has deep Rollouts visualization:

- Blue/Green preview and active status
- Canary percentage and steps
- ReplicaSet history
- Analysis results
- Rollout paused/resumed status

This is extremely useful for debugging.

---

## **6. Putting It Together – Real-World Flow**

## **Blue/Green Example**

1. Dev pushes code → Git
2. Argo CD auto-syncs → applies Rollout YAML
3. Rollouts controller creates Green version
4. QA tests Green via preview service
5. Approve promotion
6. Rollouts controller switches traffic to Green
7. Argo CD sees health = Healthy

---

## **Canary Example**

1. Git update → Argo CD sync
2. Rollouts controller:
    - shifts 5% traffic
    - runs metrics analysis
3. No issues → shifts to 20%
4. Problems → auto rollback
5. Argo CD marks health = Degraded

---

## **7. Key Interview Differences: Blue/Green vs Canary**

| Strategy | Traffic Behavior | Risk | Rollback | Use Case |
| --- | --- | --- | --- | --- |
| **Blue/Green** | Switch 100% instantly | Lower | Instant | Major updates, schema changes, controlled releases |
| **Canary** | Gradual shifting | Medium | Auto rollback with metrics | High-traffic apps, microservices, SLO-driven delivery |

---

## **8. Must-Know Interview Points**

### **1. Argo CD does NOT handle traffic routing.**

→ Rollouts controller + ingress/service mesh do.

### **2. Rollouts extends Deployment controller.**

→ Uses its own logic for RS management.

### **3. Argo CD only manages declarative state.**

→ It won’t manage the rollout timeline.

### **4. Health of Rollout is tracked via custom Lua checks.**

### **5. Git remains the source of truth for rollout strategy.**

---

## **Summary**

| Component | Responsibility |
| --- | --- |
| **Argo CD** | GitOps engine → apply Rollouts CRD, track health |
| **Argo Rollouts** | Traffic control, canary, blue/green, analysis |
| **Ingress/Service Mesh** | Weighted traffic splitting |
| **Prometheus/Datadog** | Analysis metrics |
| **Git** | Declarative strategy definition |

# **4. Argo CD App Patterns – Detailed Study Material**

---

## **1. App of Apps vs Multi-Application**

Both are valid patterns, but used for different problems.

---

### **App of Apps Pattern**

### **Definition**

You define **a single Argo CD Application** that points to a directory containing multiple **Application manifests**.

Argo CD applies them, creating child applications automatically.

### **Use Cases**

- Environment bootstrapping (installing 10–50 apps)
- Multi-cluster onboarding
- Platform engineering (install cert-manager, ingress, monitoring, logging)
- GitOps setup where Argo CD manages itself (self-managing GitOps)

### Benefits

- Centralized definition
- Fully declarative environment composition
- Easiest pattern for large-scale environments
- Reusable across clusters (just change values)

### Drawbacks

- If parent App is broken → all children blocked
- Rollout order depends on sync waves or manual logic
- Large repos may slow repo-server

---

### **Multi-Application Pattern**

### **Definition**

Each app is deployed as an independent Argo CD Application.

Applications may be created manually, via UI/CLI, or each defined individually in Git.

### **Use Cases**

- Teams managing their own application lifecycle
- Clean separation between microservices
- Independent versioning and release cadences
- When app boundaries are clear

### Benefits

- Failures isolated
- Easy per-app RBAC
- Flexible organization between teams
- No monolithic parent dependency

### Drawbacks

- Hard to orchestrate ordering at environment level
- Requires automation or templates to create many apps
- Not ideal for bootstrapping full clusters

---

### **Interview Summary**

| Pattern | Strength | Weakness |
| --- | --- | --- |
| **App of Apps** | Best for cluster bootstrapping, platform composition | Parent failure impacts children |
| **Multi-Application** | Independent microservice ownership | No natural orchestration |

---

## **2. Monorepo vs Polyrepo Structure**

Argo CD works with both repo models.

Pick based on team structure, not tools.

---

### **Monorepo**

### **Definition**

All apps, configs, Helm charts, overlays live in a **single Git repo**.

### Benefits

- Strong consistency across environments
- Single source of truth for infra + apps
- Easier to do:
    - global search/replace
    - refactoring
    - shared Helm library charts
- Simplifies App of Apps

### Drawbacks

- Large repo cost → slow clone & rendering
- Permissions hard to isolate
- Teams stepping on each other’s changes
- Complex CI pipelines
- Harder when multiple teams have different release cadences

Good fit for **one platform team** overseeing infra & app delivery.

---

### **Polyrepo**

### **Definition**

Each team or product maintains its own repo.

### Benefits

- Team-level autonomy
- Clear CI ownership
- Repo permissions well-scoped
- Faster clone/checkout
- Scales better in large organizations

### Drawbacks

- Harder to ensure consistency across repos
- More boilerplate duplicated
- Cross-repo dependencies must be well-managed
- Bootstrapping via App of Apps requires referencing many repos

Great for **enterprise/multi-team environments**.

---

### **Interview Summary**

| Option | Pros | Cons |
| --- | --- | --- |
| **Monorepo** | consistency, easier for platform mgmt | scalability & permission issues |
| **Polyrepo** | autonomy, clean boundaries | duplicated logic, inconsistent standards |

---

## **3. Helm Charts as Applications**

Argo CD natively supports Helm charts.

### Two ways:

### **(1) Helm chart in Git**

```
source:
  repoURL: ...
  path: charts/my-app
  helm:
    values: values-prod.yaml

```

### **(2) Helm chart from a Helm registry**

```
source:
  chart: nginx
  repoURL: https://charts.bitnami.com/bitnami
  targetRevision: 1.4.3

```

### Benefits

- Native integration with Helm templating
- Full values management (merge layers)
- Automatic diffing of rendered charts
- Hooks supported (pre/post sync)

### Best Practices

- Keep environment-specific values outside chart (values-prod.yaml)
- Use Helm chart versioning seriously — this is your API for deployments
- Avoid giant values.yaml merging logic → prefer overlays

### Interview Tip

Argo CD renders Helm charts using **repo-server** only. No `helm install` binary on client.

## **4. Kustomize Overlays**

Argo CD supports Kustomize natively with zero setup.

### Typical Directory Layout

```
.
├── base/
│   ├── deployment.yaml
│   └── kustomization.yaml
└── overlays/
    ├── dev/
    ├── stage/
    └── prod/

```

### Benefits

- Clean separation of base vs environment overrides
- Simple YAML patches
- Great for non-templated apps
- Builds are deterministic

### Argo CD Features

- Supports Kustomize build
- Supports Kustomize + Helm patches
- Supports Kustomize + SOPS (via KSOPS plugin)
- Supports multiple overlays in the same Application with `directory.exclude`

### Best Practices

- Limit overlays per environment (avoid overlays for overlays)
- Keep base as generic as possible
- Avoid overusing strategic merge patches → use JSON patch where possible

## **5. Jsonnet Support**

Jsonnet is a data templating language for deeply configurable manifests.

### Why Jsonnet?

- Advanced templating
- Conditional logic
- Functions + reusable libraries
- Generates pure YAML for K8s

### How Argo CD Uses Jsonnet

```
source:
  repoURL: ...
  path: my-app-jsonnet
  jsonnet:
    extVars:
      env: prod

```

### Jsonnet Ecosystem

- Works well for platform stacks
- Great for large scale operators generating many manifests
- Common in SRE-heavy orgs (Google, some fintech)

### Downsides

- Harder for average YAML developers
- Debugging can be painful
- IDE tooling weaker than Helm/Kustomize

### When to choose Jsonnet

- You need complex logic
- You want DRY, functional manifest generation
- Org relies on Jsonnet tooling already (rare)

# **5. GitOps Automation — Detailed Study Material (Argo CD)**

GitOps automation in Argo CD focuses on *making Kubernetes state converge automatically* with what you define in Git. These features reduce manual interventions, enforce consistency, and enable progressive, safe deployments.

---

## **1. Auto-Sync**

### **What It Is**

Auto-sync means Argo CD will automatically apply changes to the cluster whenever it detects a commit or state drift.

It’s the core of GitOps automation.

### **How It Works**

- Argo CD constantly compares:
    - **Git desired state**
    - **Live cluster state**
- On any difference, it triggers a reconciliation → apply manifests.

### **Modes**

There are three sync behaviors controlled by `syncPolicy`:

| Feature | Description |
| --- | --- |
| **automated:** true | Enables automatic sync |
| **prune:** true | Removes resources deleted from Git |
| **selfHeal:** true | Fixes drift even when Git hasn’t changed |

### **Example**

```yaml
spec:
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

```

### **Best Practices**

- Enable auto-sync for stable workloads.
- Disable auto-sync for critical infra (DBs, operators) unless tested thoroughly.
- Use PR-based workflow to ensure code-review before auto-syncs push changes.

---

## **2. Self-Heal**

### **What It Is**

Self-heal ensures Argo CD *reverts any manual kubectl change* that makes the live state drift from Git.

### **Use Cases**

- Accidental `kubectl delete`
- On-call engineer patches a Deployment manually
- Config drift caused by controllers

### **How It Works**

- Controller detects live → Git mismatch
- If `automated.selfHeal: true`, it immediately re-applies Git version.

### **Important Note**

Self-heal does **not** trigger on source repo changes; it only reacts to *cluster drift*.

---

## **3. Sync Windows**

### **What They Are**

Time-based rules that *allow or deny auto-sync* during certain periods.

Useful when:

- You want freeze windows (e.g., business hours)
- You support regions across timezones
- High-risk workloads should sync only during maintenance windows

### **Defined In**

AppProject (`spec.syncWindows`)

### **Example**

```yaml
spec:
  syncWindows:
    - kind: allow
      schedule: "Mon-Fri 23:00-06:00"
      duration: 7h
      applications: ["*"]

```

### **Window Types**

- **allow** window → sync allowed only in this period
- **deny** window → sync forbidden during this period

### **Override**

Manual sync always works unless `manualSync: false`.

---

## **4. Sync Waves & Order of Resources**

### **Why They Exist**

Kubernetes doesn't guarantee resource creation order.

But GitOps often requires order, such as:

- CRDs before CRs
- Namespaces before workloads
- Secrets before pods
- Ingress after services

### **How Sync Waves Work**

Argo CD assigns a **wave number** (default = 0) to each resource.

Resources are applied wave-by-wave in ascending order.

Wave is set using `argocd.argoproj.io/sync-wave`.

### **Example**

CRD first, CR next:

```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "-1"

```

Deployment:

```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "1"

```

### **Behavior**

- The lower the wave → earlier execution.
- All objects in a wave must be healthy before next wave starts.
- Supports *dependency orchestrations* without hacks.

---

## **5. Hooks (PreSync, Sync, PostSync, etc.)**

Hooks are "one-shot" tasks that run during sync operations.

### **Hook Types**

| Hook | Purpose |
| --- | --- |
| **PreSync** | Runs before any resource sync (DB migrations, checks) |
| **Sync** | Overrides normal resource apply steps |
| **PostSync** | Runs after sync completes (cleanup) |
| **SyncFail** | Runs if sync fails |
| **PostSyncFail** | Runs if post-sync hook fails |

### **Define Hook**

```yaml
metadata:
  annotations:
    argocd.argoproj.io/hook: PreSync

```

### **Hook Delete Policy**

Controls resource cleanup:

| Policy | Meaning |
| --- | --- |
| HookSucceeded | Delete hook if successful |
| HookFailed | Delete if failed |
| BeforeHookCreation | Replace old hook before creating new |

Example:

```yaml
annotations:
  argocd.argoproj.io/hook-delete-policy: HookSucceeded

```

### **Typical Use Cases**

- DB schema migrations
- Preflight checks
- Canary analysis scripts
- Cache warming
- Traffic switching with service meshes

---

## **6. Resource Tracking**

Argo CD tracks resources based on:

### **Tracking Methods**

| Method | Description |
| --- | --- |
| **Annotation** (default) | Uses annotations like `argocd.argoproj.io/instance` |
| **Label** | Uses labels for tracking |
| **Revision label** | Tracks per-Git revision |
| **Git SHA** | For deterministic tracking |

### **Why Tracking Matters**

- Avoids orphaned resources
- Ensures drift detection
- Allows prune accuracy
- Prevents Argo CD from touching “foreign” resources

### **Failure Scenarios**

- If annotations removed manually
- If controllers overwrite metadata
- If Helm charts regenerate names

Argo CD can be configured to detect this using `resourceTrackingMethod` in Projects.

---

## **7. Automated vs Manual Sync**

### **Manual Sync**

- Default
- Argo CD detects changes but doesn’t apply them
- You must click **SYNC** or run:
    
    ```
    argocd app sync <app>
    
    ```
    

### **Auto Sync**

- Uses `automated` block
- Good for: microservices, stateless apps
- Risky for: CRDs, Operators, Databases

### **Best Practice**

Use manual for infra.

Use auto for apps behind strong CI and peer review.

---

## **8. Sync Options (Prune, SelfHeal, Replace)**

### **Prune**

Removes resources deleted in Git.

Needed for truly *declarative* infra.

**Risk:** Misconfigured Git delete → production outage.

### **SelfHeal**

Restores drift from live → Git.

### **Replace**

Replaces a resource instead of patching it.

Used when:

- Server-side apply fails
- Mutating webhooks change fields
- You need full reapply

Enable using annotation:

```yaml
argocd.argoproj.io/sync-options: Replace=true

```

Other common sync options:

| Option | Use case |
| --- | --- |
| **PrunePropagationPolicy** | Control cascading deletes |
| **CreateNamespace=true** | Auto-create namespace |
| **ApplyOutOfSyncOnly=true** | Skip unchanged |

---

## **9. Lua-Based Custom Health Checks**

Argo CD lets you define custom health scripts using Lua for CRDs so Argo CD can display meaningful Health statuses.

### **Why Needed**

CRDs do not have built-in health logic.

Example: Argo Rollout → Healthy vs Degraded

Istio VirtualService → Valid vs Invalid

### **Where Defined**

In `argocd-cm`:

```yaml
resource.customizations.health.rollouts.argoproj.io: |
  hs = {}
  if obj.status ~= nil then
    if obj.status.availableReplicas == obj.spec.replicas then
      hs.status = "Healthy"
      hs.message = "All replicas available"
    else
      hs.status = "Progressing"
      hs.message = "Waiting for replicas"
    end
  end
  return hs

```

### **What You Can Control**

- Status (Healthy, Degraded, Progressing, Suspended)
- Custom messages
- Derived logic from CRD conditions

### **Use Cases**

- Rollouts health
- Operators (Kafka, MongoDB, Prometheus)
- Gateways/VirtualServices
- Terraform controller resources

# **6. Secrets Handling in GitOps — Complete Study Material**

Managing secrets is one of the biggest challenges in GitOps because **Git is immutable, auditable, and versioned**, but **must never contain plaintext secrets**.

Below are the best practices and tools used in GitOps workflows.

---

## **1. Why Secret Handling Is Hard in GitOps**

### Core Problems

- Git repositories should be fully declarative → but secrets can't be stored in plain form.
- Kubernetes `Secret` objects use base64, which is *not secure*.
- Encrypted secrets must still be usable by Argo CD / Kubernetes.
- CI/CD needs a way to decrypt/consume secrets securely at deploy time.
- Different environments (dev/stage/prod) need segregated secrets.

### Requirements in a Good GitOps Secrets Strategy

- No plaintext secrets in Git.
- Encryption or external retrieval.
- Immutable audit trail.
- Zero-trust between Git and cluster.
- Ability to rotate secrets.
- Compatible with Helm, Kustomize, and Argo CD.

---

## **2. SOPS (Mozilla SOPS)**

SOPS is one of the most popular ways to securely store encrypted secrets *inside Git*, while still enabling GitOps workflows.

---

## **How SOPS Works**

- Encrypt/Decrypt at file or field level (YAML, JSON, ENV, INI, etc.)
- Uses a *data key* which is encrypted using cloud KMSs:
    - AWS KMS
    - GCP KMS
    - Azure Key Vault
    - HashiCorp Vault
    - PGP keys

### Example Encryption Flow

1. You encrypt a file using a KMS key.
2. SOPS stores the encrypted data key inside the file.
3. Anyone with access to the KMS key can decrypt the file.
4. You commit the encrypted file to Git.
5. Argo CD decrypts secrets using:
    - SOPS + Kustomize plugin
    - SOPS + Helm plugin

---

## **SOPS File Example**

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-pass
type: Opaque
data:
  password: ENC[AES256_GCM,data:...,iv:...,tag:...]
sops:
  kms:
    - arn: arn:aws:kms:us-east-1:123:key/abc
  encrypted_regex: '^(data|stringData)$'

```

---

## **SOPS in GitOps with Argo CD**

Argo CD natively supports SOPS via:

1. **SOPS Kustomize plugin**
2. **SOPS Helm plugin**

### Argo CD Config Example

`argocd-cm`:

```yaml
data:
  kustomize.buildOptions: "--enable-alpha-plugins"

```

Plugin config file:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ConfigManagementPlugin
metadata:
  name: sops-kustomize
...

```

---

## Pros / Cons

### **Pros**

- Secrets *stay in Git* but encrypted.
- Cloud KMS for access control.
- Git history retained.
- Environment segregation built-in.

### **Cons**

- Argo CD controller needs access to decrypt secrets → operational complexity.
- Developers must install SOPS locally.
- Requires KMS permissions managed carefully.

---

## **3. External Secrets Operator (ESO)**

ESO is the most **GitOps-friendly** secrets management solution when you want **no secrets in Git at all**.

---

## **What ESO Does**

- Pulls secrets **from external secret stores** (Vault, AWS Secrets Manager, SSM Parameter Store, GCP Secret Manager).
- Creates Kubernetes `Secrets` automatically.
- Works as a Kubernetes operator.

### Flow

1. Git contains **ExternalSecret** CR (safe, no sensitive data).
2. ESO reads the CR.
3. ESO fetches secret values from a secret provider.
4. ESO creates a Kubernetes Secret.

---

## **Example ExternalSecret**

Git:

```yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-secret
spec:
  refreshInterval: 1h
  secretStoreRef:
    kind: SecretStore
    name: aws-secrets
  target:
    name: db-secret
  data:
    - secretKey: password
      remoteRef:
        key: /prod/db/password

```

AWS Secrets Manager stores the real values.

---

## **Supported Providers**

- AWS Secrets Manager
- AWS SSM Parameter Store
- HashiCorp Vault
- GCP Secret Manager
- Azure KeyVault
- GitLab
- IBM Secrets Manager

---

## **Pros / Cons**

### **Pros**

- Nothing sensitive is ever stored in Git.
- Native multi-cloud support.
- Automatic rotation.
- Kubernetes operator manages reconciliation.

### **Cons**

- Requires the cluster to access external secret stores.
- More moving components.
- Cloud IAM misconfigurations can expose secrets.

---

## **4. Vault Plugins (HashiCorp Vault)**

Vault is the most **feature-rich** secrets manager in the industry.

### Ways Vault Integrates With GitOps:

---

### **1. Vault Agent Injector**

- Injects secrets into pod via **sidecar**.
- Application gets secrets via files or env vars.
- Git contains only annotations.

Example:

```yaml
metadata:
  annotations:
    vault.hashicorp.com/agent-inject: "true"
    vault.hashicorp.com/role: "app-role"

```

---

### **2. Vault CSI Driver**

Injects secrets using the Kubernetes CSI Secrets Store.

Example Volume:

```yaml
volumes:
  - name: vault-secrets
    csi:
      driver: secrets-store.csi.k8s.io
      volumeAttributes:
        secretProviderClass: vault-db

```

---

### **3. Argo CD + Vault Plugin**

Argo CD fetches secrets from Vault at render time.

But this is less preferred because:

- Argo CD must have Vault access.
- Secrets become visible in Argo CD’s rendered manifests.

---

### Pros / Cons

### **Pros**

- Zero plaintext secrets in Git.
- Powerful RBAC and audit logs.
- Lease/TTL and dynamic secrets.

### **Cons**

- Operational overhead (Vault HA, storage backend).
- Requires Vault expertise.
- Application-level changes needed (for agent).

---

## **5. Helm Secret Management Alternatives**

Helm charts don’t handle secrets well by default.

But several patterns exist.

---

### **1. Helm Secrets (SOPS + Helm)**

- Encrypt `values.yaml` or parts of it using SOPS.
- Decrypt during `helm template`.

Example:

```bash
helm secrets enc values.prod.yaml

```

Used with GitOps by enabling the plugin inside Argo CD.

---

### **2. Using External Secrets with Helm**

- Chart templates reference Kubernetes secrets.
- ESO handles generating the secret.
- Your chart stays clean.

---

### **3. Sealed Secrets (Bitnami)**

Another common pattern.

### Flow:

1. Encrypt Secret → generates a SealedSecret manifest.
2. Commit SealedSecret to Git.
3. Sealed Secrets Controller decrypts and creates Kubernetes Secret.

### Pros

- Secrets are safe in Git.
- Private key stays only inside the cluster.

### Cons

- Tight coupling to cluster private key → **cluster-bound**.
- Harder for multi-cluster GitOps.

---

### **4. Helm + Vault values**

Inject secrets using Vault template functions:

Example:

```yaml
password: {{ vault "secret/data/db" "password" }}

```

Used through Helmfile or CI tools.

Not recommended directly with Argo CD because:

- Argo CD would need Vault access.
- Harder to reason about deterministic manifests.

---

## **6. Comparison Table — What to Use When**

| Method | Secrets in Git? | Decrypt at | Best For | Comments |
| --- | --- | --- | --- | --- |
| **SOPS** | Encrypted | Argo CD or Dev laptop | Multi-cluster GitOps | Clean, portable |
| **External Secrets Operator** | No | Cluster operator | Cloud-native environments | Best overall |
| **Vault Agent/CSI** | No | Pod runtime | Zero-trust apps | Big enterprise setups |
| **Sealed Secrets** | Encrypted | Cluster operator | Single cluster setups | Simple, but cluster-tied |
| **Helm Secrets** | Encrypted | Argo CD | Helm-first teams | Needs plugin config |

---

## **7. Recommended Production Setup**

### If you have AWS / GCP / Azure:

**Use External Secrets Operator + cloud native secret store.**

### If you want Git encryption & simplicity:

**Use SOPS + KMS.**

### If you already use Vault enterprise-wide:

**Use Vault Agent Injector or CSI Driver.**

### If small team, single cluster:

**Use Sealed Secrets.**

# **7. Argo CD – RBAC & Security (Detailed Study Material)**

Argo CD runs inside Kubernetes, syncs manifests from Git, and applies them to clusters.

This makes **access control, SSO, and resource/namespace restrictions** absolutely critical.

This module covers **all security levers Argo CD gives you**.

---

## **1. RBAC Roles and Policies**

Argo CD uses **RBAC (Role-Based Access Control)** defined in the `argocd-rbac-cm` ConfigMap.

RBAC controls **what users can do**:

- View apps
- Sync apps
- Create/edit/delete apps
- Manage projects
- Manage clusters
- Read/write repositories
- Access logs
- Access terminal
- Override sync options

---

### **RBAC Syntax**

Argo CD uses **Casbin** RBAC syntax:

```
p, <role>, <resource>, <action>, <object>, <effect>

```

### Example: allow devs to sync apps in the "dev-project"

```
p, role:devs, applications, sync, dev-project/*, allow

```

### Role Definitions

```
g, user:jane, role:devs

```

### Common Resources

- `applications`
- `projects`
- `repositories`
- `clusters`
- `accounts`
- `certificates`
- `exec` (terminal exec into pods)
- `logs` (fetch logs)

### Common Actions

- `get`
- `create`
- `update`
- `delete`
- `sync`
- `override`
- `action/*`
- (all)

---

### **Examples of Real RBAC Policies**

### 1. **Read-Only Role**

```
p, role:readonly, applications, get, *, allow
p, role:readonly, logs, get, *, allow

```

### 2. **Allow developers to sync only their apps**

```
p, role:dev, applications, sync, team1/*, allow

```

### 3. **Allow platform team full access**

```
p, role:platform, *, *, *, allow

```

### 4. **Block exec into pods for Devs**

```
p, role:dev, exec, *, *, deny

```

### 5. **Allow Devs to see apps but not edit them**

```
p, role:dev, applications, get, *, allow
p, role:dev, applications, update, *, deny

```

---

## **2. SSO Integration (OIDC, Dex)**

Argo CD supports SSO via:

- **OIDC providers**
    - Keycloak
    - Okta
    - Azure AD
    - Google
    - GitHub
    - Ping Identity
- **Dex (identity broker built into Argo CD)**

SSO maps **identity provider groups** → **Argo CD roles**.

---

### **OIDC Integration (Most Common)**

Configure in `argocd-cm`:

```yaml
data:
  oidc.config: |
    name: Azure
    issuer: https://login.microsoftonline.com/<tenant-id>/v2.0
    clientID: <client-id>
    clientSecret: <client-secret>
    requestedScopes:
      - openid
      - profile
      - email
      - groups

```

Mapping roles:

```
g, OIDC:dev-group, role:dev
g, OIDC:platform-group, role:platform

```

---

### **Dex Integration**

Dex acts as a “broker” allowing:

- LDAP
- GitHub
- SAML
- OIDC
- Google

Most enterprises phase Dex out in favor of pure OIDC.

---

### **Why SSO Matters**

- No local passwords
- Auditable login via IdP
- Group-based access control
- Enable MFA/conditional access policies
- Deprovisioning automated

---

## **3. Repo Whitelisting**

Argo CD can **restrict which Git repos** users or app projects can use.

This avoids:

- Syncing untrusted repos
- Supply-chain attacks via malicious manifests
- Developers pointing Argo CD to personal GitHub repos

---

### **Configure in AppProject**

```yaml
spec:
  sourceRepos:
    - https://github.com/myorg/*

```

Or restrict to a single repo:

```yaml
spec:
  sourceRepos:
    - https://github.com/myorg/prod-config

```

Attempting to sync from a repo outside this list → **denied**.

---

## **4. Restrict Cluster Resources Using AppProject**

AppProject is **the main multi-tenancy control plane** in Argo CD.

Controls:

- What repos can be used
- What clusters can be deployed to
- What Kubernetes resources can be created
- What namespaces they can deploy into

---

### **Restricting Allowed Kubernetes Resources**

Block risky cluster-scoped resources:

```yaml
spec:
  clusterResourceWhitelist:
    - group: ""
      kind: ConfigMap
    - group: apps
      kind: Deployment

```

Default behavior:

- cluster-scoped resources are **DENIED** unless explicitly whitelisted
- namespaced resources are **ALLOWED** unless blacklisted

### Block specific resources

```yaml
spec:
  clusterResourceBlacklist:
    - group: rbac.authorization.k8s.io
      kind: ClusterRole

```

---

## **5. Limiting Namespace or Cluster-Scope Deployments**

This is essential for tenant isolation in multi-tenant clusters.

---

### **Restrict to Specific Namespaces**

AppProject example:

```yaml
spec:
  destinations:
    - namespace: team1
      server: https://kubernetes.default.svc

```

Apps under this project *cannot* deploy to:

- other namespaces
- other clusters

---

### **Restrict Cluster Access**

Only allow dev project to deploy to dev cluster:

```yaml
spec:
  destinations:
    - server: https://dev-cluster.k8s.mycloud
      namespace: '*'

```

Block prod cluster access:

```yaml
spec:
  destinations:
    - server: https://prod-cluster.k8s.mycloud
      namespace: "!*"   # deny

```

---

## **Security Best Practices**

### **1. Least Privilege RBAC**

- Read-only roles for QA/stakeholders
- Sync-only roles for Devs
- Full Access only for Platform/SRE

### **2. Block Dangerous Resources**

- ClusterRole
- ClusterRoleBinding
- CRD updates
- MutatingWebhookConfiguration
- ValidatingWebhookConfiguration

### **3. Restrict Git Repos**

Avoid allowing:

```
sourceRepos: ["*"]

```

### **4. Enforce Namespace Boundaries**

Use AppProjects strictly.

### **5. Enable SSO + MFA**

Never use local Argo CD password except bootstrap.

### **6. Disable Exec Into Pods (unless truly needed)**

Argo CD RBAC:

```
exec, get, deny

```

### **7. Disable Auto-Creation of Projects**

Treat AppProject like a boundary, not a suggestion.

Using `sourceRepos` in AppProject.

Using `destinations` in AppProject.

# **8. Multi-Cluster GitOps — Detailed Study Material**

Most enterprises run **multiple Kubernetes clusters**:

- dev
- stage
- prod
- performance
- DR
- regional clusters
- EKS + AKS + GKE combos

Argo CD is designed to **manage all clusters from one control plane** using GitOps.

Below is everything you need to understand for interviews and real use.

---

## **1. Managing Multiple Clusters from a Single Argo CD**

Argo CD itself runs **inside any one Kubernetes cluster**, but it can manage **many external clusters**.

### Supported cluster types:

- any CNCF-compliant Kubernetes cluster
- EKS
- AKS
- GKE
- on-prem clusters
- local clusters (kind/minikube)

---

## **How Argo CD Multi-Cluster Works**

Argo CD acts like:

- a **Git watcher**
- a **manifest renderer**
- a **Kubernetes API client** for each connected cluster

Argo CD stores:

- cluster credentials
- cluster server endpoint
- cluster name
- cluster namespace restrictions

### Where clusters are stored:

- In the Argo CD namespace inside the Secret object:
    
    ```
    Secret: argocd-cluster-<cluster-name>
    
    ```
    

---

## **Flow**

1. **App CRD** declares the cluster via `.spec.destination.server`.
2. Argo CD finds credentials in cluster Secret.
3. Argo CD validates connection.
4. Argo CD applies manifests to the remote cluster via its Kubernetes API.

---

### **Adding Clusters**

### **1. Through CLI (recommended)**

```
argocd cluster add <kube-context-name>

```

This creates:

- a Kubernetes ServiceAccount in the target cluster
- a ClusterRole + ClusterRoleBinding
- a secret `argocd-cluster-xxxxx` in the Argo CD cluster

You can add multiple clusters this way.

---

### **2. Declarative Method (fully GitOps)**

Store the cluster secret as a Kubernetes Secret:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: argocd-cluster-prod
  labels:
    argocd.argoproj.io/secret-type: cluster
type: Opaque
stringData:
  name: prod
  server: https://prod-k8s.mycloud
  config: |
    {
      "bearerToken": "xxxx",
      "tlsClientConfig":{"insecure":false,"caData":"xxx"}
    }

```

This is useful for:

- bootstrapping Argo CD itself
- fully declarative environments
- automation in IaC (Terraform/Pulumi)

---

## **2. Cluster Secrets & Kubeconfigs Stored Safely**

Since Argo CD needs **cluster admin-like access**, protecting cluster creds is critical.

---

### **Storage Pattern (default)**

Argo CD stores each cluster connection as a **Kubernetes Secret**:

Example Secret:

```
argocd-cluster-dev
argocd-cluster-prod
argocd-cluster-eu-west

```

These contain:

- API server URL
- CA cert
- Bearer token or client cert
- Namespace restrictions

---

## **Security Best Practices**

### **1. Do NOT store plain kubeconfigs in Git**

Always use native Argo CD-managed secrets in the cluster.

---

### **2. Use Secret Management Tools**

You can encrypt cluster secrets using:

- **SOPS** (recommended)
- Sealed Secrets
- External Secrets Operator

### Example with SOPS:

You store encrypted secret in Git → Argo CD decrypts → creates actual cluster secret.

---

### **3. Use Least-Privilege SA in Target Clusters**

`argocd cluster add` creates an SA with cluster-admin by default.

This is dangerous in prod.

Instead, create a restricted role:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: argocd-manager
rules:
  - apiGroups: ["*"]
    resources: ["deployments", "statefulsets", "services"]
    verbs: ["*"]

```

Argo CD needs CRUD access to resources it syncs, not full cluster-admin.

---

### **4. Encrypt etcd (if on-prem)**

Cluster secrets live in Secrets → Secrets live in etcd.

Enable:

- etcd encryption
- limited access to etcd nodes

---

### **5. Never expose Argo CD externally without SSO**

Local accounts = security risk.

---

## **3. Cluster Bootstrapping via App of Apps**

This is the most powerful pattern in GitOps at scale.

You bootstrap entire clusters using a **parent Application** that manages:

- namespaces
- Argo CD projects
- Argo CD apps
- CRDs
- core platform components
- environment-specific applications

---

### **What is the App of Apps Pattern?**

You create **one Argo CD Application**, which points to a repo containing many more `Application` manifests.

### Example:

```
root-app
└── apps
    ├── logging
    ├── ingress
    ├── monitoring
    ├── team1-apps
    └── team2-apps

```

Argo CD deploys:

- Prometheus stack
- Ingress controllers
- Cert-manager
- ECK/ELK
- app namespaces
- applications per team

All **just by syncing one Application**.

---

### **Cluster Bootstrapping Flow**

### **Step 1 — Create New Kubernetes Cluster**

Use any tool:

- EKS / AKS / GKE
- Terraform
- kubeadm

---

### **Step 2 — Install Argo CD into the Cluster OR Install Argo CD in a central cluster**

Both patterns exist:

### **Pattern A: Central Argo CD**

One Argo CD → manages all clusters.

### **Pattern B: Each cluster has its own Argo CD controller**

Used in air-gapped + regulated setups.

---

### **Step 3 — Add cluster to Argo CD**

```
argocd cluster add <context>

```

---

### **Step 4 — Create a "Root Application"**

This app is the cluster's bootstrap.

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: cluster-bootstrap
spec:
  project: default
  destination:
    server: https://prod-cluster.example.com
    namespace: argocd
  source:
    repoURL: https://github.com/myorg/clusters
    path: prod/env/bootstrap
  syncPolicy:
    automated:
      prune: true
      selfHeal: true

```

---

### **Step 5 — Root App syncs and creates children apps**

Example directory:

```
bootstrap/
  apps/
    ingress.yaml
    cert-manager.yaml
    monitoring.yaml
    apps-team1.yaml
    apps-team2.yaml

```

Each file is another Argo CD Application → pointing to its own Git path.

---

### **Step 6 — Cluster bootstraps itself fully**

Within minutes:

- CRDs installed
- Namespaces created
- Ingress installed
- Grafana/Prometheus deployed
- All applications deployed
- Secrets provisioned (ESO/SOPS/etc.)
- RBAC templates applied

This pattern enables **zero-touch cluster creation**.

---

## **App of Apps — Best Practices**

### **1. Split infra and apps**

```
apps/infra/
apps/services/

```

### **2. Avoid cyclic dependencies**

Parent apps never reference each other.

### **3. Use Sync Waves**

Ensure:

- CRDs install first
- Operators install next
- Application workloads install last

Example:

```yaml
metadata:
  annotations:
    argocd.argoproj.io/sync-wave: "-1"

```

---

## **Multi-Cluster Architecture Patterns (Interview-Ready)**

### **Pattern 1 — Centralized Argo CD (most common)**

- One Argo CD instance
- Manages 10–500 clusters
- High-availability Argo CD
- Uses AppProject for multi-tenancy

Pros:

- Single control plane
- Simple
- Good for large orgs

---

### **Pattern 2 — One Argo CD per Cluster**

Used when:

- Clusters are air-gapped
- Regulatory isolation needed
- Multi-tenant cluster hosting prohibited

Cons:

- Hard to manage updates
- More overhead

---

### **Pattern 3 — Hub-Spoke**

- One central Argo CD manages only infra/cluster stuff
- Each cluster has its app-level Argo CD instance

Good for platform teams that don’t want to manage workloads.

---

## **Interview Cheat Sheet (Rapid Answers)**

1. **Q: How does Argo CD manage multiple clusters?**
    
    By storing each cluster’s credentials as a Secret of type `argocd.argoproj.io/secret-type: cluster`, and using the cluster API endpoint in the App spec destination.
    
2. **Q: How do you store cluster credentials securely?**
    - Use SOPS/Sealed Secrets/ESO
    - Use least-privileged service accounts
    - Encrypt etcd
    - Avoid plaintext kubeconfigs in Git
3. **Q: How do you bootstrap a cluster the GitOps way?**
    
    Using **App of Apps** with a “root app” that deploys:
    
    - CRDs
    - platform components
    - team apps
    - policies
4. **Q: What’s the difference between centralized and distributed Argo CD?**
    
    Centralized = 1 Argo CD for all clusters
    
    Distributed = 1 Argo CD per cluster
    
    Choose based on compliance, latency, air gap, scale.
    
5. **Q: How do you ensure environment separation?**
    
    Use AppProjects + per-environment repos + per-cluster destinations.

# **10. Argo CD – Performance & Scaling**

Argo CD is powerful but **performance can degrade** in large organizations with:

- hundreds of apps
- large Helm charts
- multi-cluster deployments
- frequent Git commits

This module covers **bottlenecks, scaling points, and optimization techniques**.

---

## **1. When to Scale `repo-server`**

The **repo-server** is responsible for:

- Cloning Git repositories
- Rendering manifests (Helm, Kustomize, Jsonnet)
- Serving manifests to the Application Controller

---

### **Signs repo-server needs scaling**

- Slow app sync times for multiple apps
- CPU/memory spikes in repo-server pods
- Large manifests (>1–5MB per app) causing timeouts
- Multiple Helm charts or Kustomize overlays per app

---

### **Scaling Recommendations**

1. **Horizontal Pod Autoscaling (HPA)**
    - CPU-based scaling for repo-server pods
    - Example:
    
    ```yaml
    apiVersion: autoscaling/v2
    kind: HorizontalPodAutoscaler
    metadata:
      name: argocd-repo-server
    spec:
      scaleTargetRef:
        apiVersion: apps/v1
        kind: Deployment
        name: argocd-repo-server
      minReplicas: 2
      maxReplicas: 6
      metrics:
        - type: Resource
          resource:
            name: cpu
            target:
              type: Utilization
              averageUtilization: 70
    
    ```
    
2. **Increase resources**
    - CPU: 2–4 cores
    - Memory: 4–8 GB
3. **Shard repos**
    - For very large organizations, run **multiple repo-servers** per set of apps.

---

## **2. Large Helm Charts**

Large charts can slow down Argo CD because repo-server has to:

- Render templates
- Resolve dependencies
- Merge values.yaml / override values

---

### **Performance Tips**

- Use `-helm-set` to override small values instead of massive values files
- Avoid rendering unnecessary subcharts
- Use **helm 3 lazy-loading** to reduce memory footprint
- Consider **pre-rendering charts** for extremely large apps (CI pipeline renders, Argo CD applies pre-rendered manifests)

---

## **3. Manifest Size Issues**

### **Problems**

- Large manifests (>1–5 MB) can:
    - Cause syncs to timeout
    - Crash repo-server
    - Slow diffing operations

### **Solutions**

1. **Split large apps into multiple smaller apps**
    - Example: one Helm chart per microservice
2. **Use `App of Apps` pattern**
    - Avoid one monolithic app with hundreds of resources
3. **Reduce unnecessary manifests**
    - Remove unused CRDs or ConfigMaps
4. **Increase `application.instanceLabelKey`**
    - Helps with resource tracking without reading entire manifest in memory

---

## **4. Caching and Concurrency**

Argo CD uses caching and concurrency to improve performance.

---

### **Caching**

1. **Redis cache** stores:
    - Manifest diff results
    - Application status
    - Resource health status
2. **Configuration**

```yaml
data:
  resource.customizations.health.timeoutSeconds: "30"
  redis.resourceCacheTTL: "3m"

```

### **Concurrency**

- Argo CD syncs multiple apps concurrently
- Controlled by:
    - `controller.concurrencyLimit`
    - Default: 5
- Increase for **large orgs with hundreds of apps**
- Be mindful of cluster API server limits (don’t overload K8s API)

---

## **5. Best Practices for Large Organizations**

1. **App Segmentation**
    - Use **App of Apps** to split environments
    - Use **projects per team/namespace**
2. **Use HPA for critical components**
    - repo-server
    - application-controller
    - Redis
3. **Optimize repository structure**
    - Smaller repos → faster clones
    - Monorepo → slower sync, consider shallow clones
4. **Limit Diff Scope**
    - Use `ignoreDifferences` for resources with frequent external changes
    - Example: `status`, `last-applied` annotations
5. **Leverage Helm/Kustomize pre-rendering**
    - Offload heavy computation to CI pipeline

## **6. Common Scaling Patterns**

| Pattern | When to Use | Pros | Cons |
| --- | --- | --- | --- |
| Multiple Repo-Servers | Hundreds of apps, large manifests | Parallel rendering | More resources, more HA setup |
| Shallow Git Clones | Large Git repos | Faster clone, less memory | Cannot access full Git history easily |
| App of Apps | Monolithic deployments | Manage hundreds of apps | Slight complexity in bootstrapping |
| Pre-render Helm | Very large charts | Reduces repo-server load | Extra CI step, loses Argo CD dynamic rendering |
| Sync Windows | Multi-cluster deployments | Reduce API server overload | Less real-time drift correction |

## **8. Summary**

**Performance Bottlenecks:**

1. repo-server under heavy load (large charts, many apps)
2. Application-controller computing health and diffs
3. Large manifest sets (>1–5 MB)
4. Multi-cluster concurrency overloading API servers

**Scaling Actions:**

- Horizontal scaling of repo-server
- Increase concurrency limits carefully
- Split apps / repos
- Use App of Apps pattern
- Pre-render manifests for heavy Helm charts
- Use caching (Redis) efficiently
- Monitor, alert, and tune continuously

**Goal:**

Make Argo CD **highly available, responsive, and safe for large-scale GitOps deployments**, without overloading clusters or causing sync failures.
